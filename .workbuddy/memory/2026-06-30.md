## 通讯沙盘校验工具开发

**需求**：基于校规文档，开发一款离线运行的沙盘数据校验工具，支持 Mac/Windows。

**产出**：`sandbox_validator.py` — 单文件 GUI 应用（tkinter + openpyxl）

**校验范围**：
- 通讯通路门店（67列）：枚举/数值/编码/引用/勾稽
- 通讯客户沙盘（58列）：枚举/数值/渠道/门店数/促销员勾稽

**核心校验规则**：
- 必填字段检查
- 枚举值校验（省份/进度/状态/类型/运营商 等）
- 整数范围校验（0-999999）
- 手机号格式校验（11位1x开头）
- 门店编码格式（字母/6开头/纯数字≥8位）
- 地理编码/商圈编码引用存在性校验
- 渠道编码格式（纯数字或'无'）
- 价位段求和 = 品牌求和
- 专卖店数 ≤ 总门店数
- 促销员自有 + 厂家 = 总数

**性能**：只读模式加载 + 写模式写入分离，单文件 ~1.9s 处理 5700 行数据。

**输出**：错误写入原表校验列（红字标记），可导出 TXT 报告。

## PassWall2 nftables 规则丢失修复

**问题**：外网（科学上网）不通，国内正常。

**根因**：PassWall2 的 `inet passwall2` nftables 表和 `table 999` 策略路由丢失。xray/chinadns-ng 进程正常但流量没有经过透明代理。触发原因可能是每小时一次的规则/订阅更新导致 dnsmasq 重启。

**修复**：
```bash
nft -f /tmp/etc/passwall2/PSW2_RULE.nft
ip route add local default dev lo table 999
```

**自动化**：部署了 `/root/fix_passwall2.sh` + cron 每 5 分钟执行，自动检查和恢复：
- nftables passwall2 表
- 策略路由 table 999
- xray 进程
- chinadns-ng 进程
- DNS 配置（多服务器 + 缓存）

## DNS 优化

### 缓存层级
| 层 | 优化前 | 优化后 |
|----|--------|--------|
| 主 dnsmasq | cache-size=0 | cache-size=1500 |
| dnsmasq_default | cache-size=0 | cache-size=2000 |
| chinadns-ng | 无缓存 | cache=4096, stale=3600s, pre-refresh=30% |

### DNS 上游
- **china-dns**：223.5.5.5, 223.6.6.6, 119.29.29.29（从 1 个增加到 3 个，并发查询）
- **主 dnsmasq fallback**：同样增加到 3 个
- 清理了 chinadns-ng 冗余的 `filter-qtype 65`

### DNS 架构
```
设备 → dnsmasq(:53) → redirect → dnsmasq_default(:11400) → xray DNS(:15353) → chinadns-ng(:15354) → 上游
```
